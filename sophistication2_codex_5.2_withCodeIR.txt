    | This is the ORM “bulk UPDATE vs. in-session state” trap, and it’s usually caused by running the bulk UPDATE while your per-object changes are still **unflushed**.
    | 
    | - `Query.update()` builds a Core `UPDATE` and runs it via `Session.execute(...)` with `execution_options={"synchronize_session": ...}` (`lib/sqlalchemy/orm/query.py:3282`). The SQL `WHERE` is evaluated **in the database**, against **database row values**, not your in-memory object state.
    | - Before executing, SQLAlchemy *tries* to autoflush in `bulk_persistence._BulkUDCompileState.orm_pre_session_exec()` (`lib/sqlalchemy/orm/bulk_persistence.py:757-760`), but that flush goes through `Session._autoflush()` which only flushes if `session.autoflush` is currently true (`lib/sqlalchemy/orm/session.py:3112-3114`). If you’re inside `session.no_autoflush` (or `Session(autoflush=False)`), the flush is skipped.
    | - With no flush, the DB still has the “old” values, so rows you *think* don’t match (based on your pending in-memory edits) can still match the `WHERE` and get updated.
    | - Then `synchronize_session` kicks in and may **expire/overwrite attributes on matched in-session objects**, specifically expiring attributes that were previously modified when autoflush is off (`lib/sqlalchemy/orm/bulk_persistence.py:1917-1922`). That can effectively discard your pending per-object changes before the later commit flush ever writes them.
    | - On `commit()`, the session expires instances by default (`expire_on_commit=True`), so you immediately “see” the database state after commit (`lib/sqlalchemy/orm/session.py:1136-1139`)—making it look like the bulk update overwrote your edits (because it did, and/or your edits were expired away).
    | 
    | What to do:
    | - Ensure the relevant changes are flushed **before** the bulk UPDATE: call `session.flush()` (or don’t use `no_autoflush` around the bulk DML).
    | - If you must bulk-update without synchronizing state, use `synchronize_session=False` and then `session.expire_all()` / refresh affected objects—but still flush first if the `WHERE` should reflect your in-memory edits.
    | - Best practice: don’t mix pending ORM edits and ORM bulk DML against the same rows in one unit of work unless you control the flush boundary explicitly.

    | It would “work” in the sense that `lazy="selectin"` is a supported loader strategy, but changing the **default** from `select` → `selectin` would be a major behavioral change and would break a lot of assumptions (and this repo’s tests).
    |

    | - The docstring for `relationship()` also makes clear that `selectin` is **eager** (“loaded *as the parents are loaded*”), while `select` is **lazy** (“loaded lazily when the property is first accessed”): `lib/sqlalchemy/orm/_orm_constructors.py:1465`. This isn’t just an optimization; it changes *when* SQL is emitted.
    | 
    | **What the code actually does**
    | - `lazy="select"` uses `_LazyLoader` (`lib/sqlalchemy/orm/strategies.py:695-700`) which installs per-instance callables and emits SQL only on attribute access (`_LazyLoader.create_row_processor` in `lib/sqlalchemy/orm/strategies.py:1177`, `_LazyLoader._load_for_state` in `lib/sqlalchemy/orm/strategies.py:889`). It also has the many-to-one identity-map “`use_get`” optimization.
    | - `lazy="selectin"` uses `_SelectInLoader` (`lib/sqlalchemy/orm/strategies.py:2951-2952`) which registers a `_PostLoad` callable during result processing (`_SelectInLoader.create_row_processor` calls `_PostLoad.callable_for_path` at `lib/sqlalchemy/orm/strategies.py:3130`).     | 
    | **Concrete breakage in this codebase**
    | - Multiple tests assert that the *default* relationship loader has `strategy.use_get` (i.e., they are implicitly depending on the default being `_LazyLoader`), e.g. `test/orm/test_lazy_relations.py:1748` and `test/orm/declarative/test_tm_future_annotations_sync.py:336`. With `lazy="selectin"` as default, those assertions would fail (and in many places the attribute/behavior wouldn’t even exist).
    | 
    | **Real-world behavioral/compat risks if default changed**
    | - Queries that currently load only parent rows would now emit additional SQL for *every* relationship unless eagerloads are disabled; this changes query counts, timing, and transactional behavior.
    | - Relationships would be populated even if never accessed (memory/time regressions are common in “large graph” models).
    | - `selectin` relies on `IN` lists and may need fallback paths when FK values aren’t present (`_load_for_path` explicitly “degrades” if FK cols aren’t loaded: `lib/sqlalchemy/orm/strategies.py:3175-3180`), so it can change SQL shapes and performance characteristics.
    | 
    | Net: flipping the default from `select` to `selectin` would be a backwards-incompatible change (semantics, performance profile, and test expectations). If the goal is “avoid N+1 by default”, it likely needs to be an opt-in configuration knob rather than changing `relationship()`’s default.
